# Contracts Publication Model

This document records the split between private development and public contract
publication for the v3 contracts work.

## Repositories

### Private monorepo

Path in this workspace:

- `<private-monorepo>`

Purpose:

- primary development workspace for contracts, deployment automation, frontend,
  indexer, tests, local `.env` files, and internal rollout notes
- internal test deployments and local validation
- application-specific post-deployment operations, such as syncing frontend or
  indexer env files from deployment state

This repository remains the source of truth for active implementation work.

### Public contracts repository

Expected local path:

- `<public-contracts-repo>`

Purpose:

- public Solidity source publication
- public ABI publication
- public deployment and verification tooling
- GitHub Actions based real deployment to protected environments
- release artifacts such as `address-book.yml` and
  `verification-manifest.json`
- public documentation written for verifiers, integrators, and users who do not
  know the private monorepo details

The public repository is not the primary development workspace. It should not
contain private product docs, local env synchronization scripts, post-deployment
content automation, Hardhat test harnesses, or site/indexer implementation
details unless they directly affect contract verification.

## Deployment Model

Development and test validation may happen in the private monorepo.

Real externally relevant deployments should happen from the public contracts
repository once a contract version is ready to publish. That gives the public
repository a clean chain of custody for:

- the Solidity sources used for deployment
- the vendored QAN bundler
- the Node `solc + ethers` deployment pipeline
- the automated audit evidence bundle
- the generated address book
- the generated verification manifest
- workflow artifacts and releases

GitHub Actions variables/secrets for the public repository should be configured
per protected environment. `PRIVATE_KEY` is the only required secret in the
current model. Non-sensitive values such as `RPC_URL`, owner addresses, fee bps,
`EVM_VERSION`, and optional `NETWORK_NAME` should be environment variables unless
they include credentials.

## Public Sync Surface

Only a curated surface should be copied from private to public by default:

- `blockchain/contracts/*.sol` for deployed qbitmarket contracts
- `blockchain/contracts/abis/*.json`
- `blockchain/contracts/abis/*.ts`
- `deployment/scripts/bundle_contracts.cjs`
- `deployment/scripts/bundle_contracts.sh`

Deployment runtime files such as `deployment/app/deploy.js`, `statePaths.js`,
`describe_verification.js`, `Dockerfile`, `entrypoint.sh`, and package files may
be ported to the public repository only after a public-scope review. The private
versions can legitimately contain internal env syncing or deployment-adjacent
features that should not be copied automatically.

Do not sync:

- real `.env` files
- `deployment/updateEnv.js`
- `deployment/env/`
- app-specific post-deployment scripts such as role, collection, set, or token
  automation
- private docs or session notes
- frontend or indexer code
- Hardhat config, tests, mocks, or local-only helper contracts
- generated collection/set JSON data
- `multicall3` unless qbitmarket explicitly deploys and verifies it as part of a
  public release

Bundled contracts are deployment artifacts. They may be published as release
artifacts or generated by the public workflow, but they should not be treated as
the human-edited source of truth.

## Public Audit Evidence

The public repository should include a dedicated audit-evidence workflow before
externally relevant secondary-market rollout.

Intent:

- make the audit process reproducible by third parties
- attach machine-readable reports to the exact public commit used for deployment
- show that important findings were fixed, tested, or explicitly triaged before
  release
- avoid implying that automated tools are a full formal audit or a guarantee of
  absence of vulnerabilities

Recommended workflow shape:

- run compile and the release test suite with dummy keys and a writable `TMPDIR`
- run Slither on the raw source entrypoints with generated bundles and
  dependencies filtered out
- run Mythril or equivalent symbolic analysis against compiled bytecode artifacts
  so the workflow does not depend on source-level compiler downloads inside the
  analysis container
- generate and upload artifacts containing:
  - tool versions
  - source commit SHA
  - compiled artifact hashes
  - bundled deployment source hashes
  - an executive security verification summary as Markdown, HTML, and PDF
  - Slither JSON reports
  - Slither accepted-finding policy and triage-gate output
  - Mythril JSON reports
  - the human triage document
  - the invariant-test rule document
  - the lifecycle edge-case regression tests
  - the verification manifest when tied to a deployment run

Release policy:

- unresolved high-confidence/high-impact findings should fail the workflow
- accepted informational findings may remain only when they are documented in
  the triage file with rationale and, where possible, regression coverage
- new Slither detector families or impact levels fail the triage gate until
  they are fixed or explicitly added to
  `docs/slither-accepted-findings.json`
- the final public release should link to the audit-evidence artifact and the
  deployment verification manifest

## Sync Command

Use the private helper script from the monorepo root:

```bash
scripts/export_public_contracts.sh \
  --public-repo <public-contracts-repo>
```

The command is dry-run by default. To apply the default safe surface:

```bash
scripts/export_public_contracts.sh \
  --public-repo <public-contracts-repo> \
  --apply
```

If a deployment tooling change is intentionally public-safe, include it
explicitly:

```bash
scripts/export_public_contracts.sh \
  --public-repo <public-contracts-repo> \
  --include-deployment-tooling \
  --apply
```

Use that flag sparingly. The public repository has its own intentionally scoped
workflow/docs/scripts, and those should not be overwritten from the private
monorepo by default.

When the audit-evidence surface is intentionally ready for publication, include
it explicitly:

```bash
scripts/export_public_contracts.sh \
  --public-repo <public-contracts-repo> \
  --include-audit-evidence \
  --apply
```

This copies the public audit-evidence workflow, Slither/Mythril helper scripts,
the manifest and executive-summary generators, the triage and invariant-test
documents, the minimal Hardhat runtime files, the Slither accepted-finding
policy, and the focused reentrancy, invalidation, payment-token policy,
invariant, and lifecycle edge-case tests used to justify the
automated-audit findings.

Then review the public repository before committing:

```bash
git -C <public-contracts-repo> status --short
git -C <public-contracts-repo> diff --stat
git -C <public-contracts-repo> diff
```

The script intentionally refuses to write into a dirty public repository unless
`--allow-dirty` is passed. Prefer committing or stashing public changes first.

## Release Checklist

Before publishing or deploying from the public repository:

1. Finish and verify the contract change in the private monorepo.
2. Run the export script in dry-run mode.
3. Apply the export into the public repository.
4. Review the public diff for accidental private details.
5. Run the public `Contract CI` workflow or equivalent local checks.
6. Commit and push the public repository.
7. Deploy from the public `Contract Operations` workflow in the target protected
   environment. For partial deploys that reuse existing addresses, pass the
   prior deploy workflow run ID through `deploymentStateRunId` so the workflow
   can download the previous `address-book.yml` before submitting transactions.
8. Run the workflow bytecode verification operation, or rely on the deployment
   run's built-in on-chain bytecode validation, before promoting the address
   matrix. For standalone verification-only workflow runs, pass the prior deploy
   workflow run ID through `deploymentStateRunId` so the workflow can download
   the `contract-<env>-deploy-state` artifact containing
   `verification-manifest.json`. Deploy runs validate the aliases produced by
   their current deploy target selection; standalone verification runs validate
   the whole manifest unless a `verificationContract` filter is supplied.
9. Download and retain the generated `address-book.yml`,
   `contracts.auto.tfvars.json`, `verification-manifest.json`, and ABI
   artifacts.
10. Use the generated state in the private monorepo for frontend/indexer env
   syncing, but keep those consumer-specific scripts private.
11. Create an immutable public release when the deployment should become a
    durable verification reference.

## Next Session Resume Note

When continuing v3 implementation, treat this split as a standing constraint:

- implement and test in the private monorepo
- export only the curated public surface
- deploy externally relevant contract releases from the public repository
- never publish private app/deployment convenience scripts by default
